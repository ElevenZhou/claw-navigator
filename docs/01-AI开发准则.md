## 第二章：开发红线（绝对不可违反）

### 2.1 产品红线
1. **本站定位是 Claw 系产品导航与对比站**，不是通用AI工具导航、不是论坛、不是商城
2. **不得添加任何商业化功能**（广告、付费、会员、充值）
3. **不得添加用户登录/注册系统**（MVP阶段）
4. **不得添加UGC功能**（评论、投稿、评分）
5. **产品数据必须客观中立**，不得对任何产品做主观贬低或过度吹捧

### 2.2 技术红线
1. **技术栈锁定为 Astro + Tailwind CSS**，不得自行更换为 Next.js/Nuxt/Vue等
2. **数据存储使用 JSON 文件**，不得引入数据库（MySQL/MongoDB/SQLite等）
3. **不得引入大型UI库**（Ant Design/Element UI/Material UI等），只允许Tailwind + 少量手写CSS
4. **交互组件使用 React Islands**，不得引入 Vue/Svelte 等其他框架
5. **构建输出必须是纯静态文件**，不得有服务端运行时依赖

### 2.3 安全红线
1. 不得在前端代码中硬编码任何密钥/Token/API Key
2. 不得使用 `eval()`、`innerHTML`（除非数据经过sanitize）
3. 不得引入任何远程JS脚本（analytics待Phase 2后再加）
4. 所有用户输入（搜索框）必须做 XSS 防护

## 第三章：代码规范

### 3.1 文件命名

```
✅ 正确：
src/components/ProductCard.astro
src/components/CompareTable.tsx
src/data/products.json
src/pages/product/[id].astro

❌ 错误：
src/components/product-card.astro      （禁止kebab-case）
src/components/Product_Card.astro      （禁止下划线）
src/data/Products.json                 （禁止大写开头）
```

**规则**：组件文件 PascalCase，数据/工具文件 camelCase，页面文件 kebab-case。

### 3.2 组件规范

```astro
---
// 1. frontmatter 顶部，先导入再声明props
import ProductCard from '../components/ProductCard.astro';

interface Props {
  title: string;
  products: Product[];
}

const { title, products } = Astro.props;

// 2. 数据处理逻辑
const filtered = products.filter(p => p.status === 'active');
---

<!-- 3. 模板区，使用Tailwind类 -->
<div class="grid grid-cols-3 gap-6">
  {filtered.map(product => (
    <ProductCard product={product} />
  ))}
</div>

<style>
  /* 4. 仅当Tailwind无法满足时才用 scoped style */
</style>
```

### 3.3 React Island 规范

```tsx
// 交互组件必须标记 client:* 指令
// ✅ 正确
<ProductCard client:visible />     // 进入视口时加载
<CompareTable client:load />       // 页面加载时立即加载
<SearchBox client:idle />          // 浏览器空闲时加载

// ❌ 错误
<ProductCard client:only="react" /> // 过度加载
```

### 3.4 TypeScript 类型定义

```typescript
// src/types/product.ts — 所有类型必须集中管理
```

## 第四章：Git 工作流

### 4.1 分支策略（MVP阶段简化）
```
main        ← 生产分支，始终保持可部署
  └── dev   ← 开发分支，日常开发在此
```

### 4.2 Commit 规范
```
<type>(<scope>): <subject>

type 枚举：
  feat     — 新功能
  fix      — Bug修复
  style    — 样式调整（不影响逻辑）
  refactor — 代码重构（不改变功能）
  docs     — 文档更新
  chore    — 构建/工具/依赖变更

示例：
feat(home): 添加产品卡片组件
fix(compare): 修复对比表横向滚动问题
style(global): 调整导航栏间距
docs(readme): 更新项目说明
```

### 4.3 禁止事项
- 禁止直接推送到 main 分支
- 禁止提交 node_modules、dist、.astro 等生成目录
- 禁止提交包含密钥/Token的文件
- 禁止使用 `--force` 推送

## 第五章：AI 协作协议

### 5.1 上下文传递
当开发会话中断或切换AI时，新AI必须：
1. 先阅读本准则文档 + 技术框架文档 + 项目开发文档
2. 读取 `src/data/products.json` 了解数据结构
3. 检查当前代码状态（已完成的文件列表）
4. 从上次中断点继续，不重复已完成的工作

### 5.2 进度报告
每完成一个功能模块，AI必须：
1. 更新本文档末尾的「开发进度表」
2. 列出已完成的文件清单
3. 标注当前阶段和下一步计划

### 5.3 异常处理
当遇到以下情况时，AI必须**暂停并请求确认**，不得自行决策：
- 需求文档有歧义或冲突
- 技术方案无法满足需求
- 发现安全风险或合规问题
- 依赖包存在已知漏洞

## 第六章：质量检查清单

### 每个文件提交前必须自检

```
□ 文件命名符合规范
□ 组件有顶部注释
□ TypeScript 无 any 类型（除非必要且有注释说明）
□ 无 console.log / debugger 残留
□ Tailwind 类名无拼写错误
□ 颜色值使用设计令牌或统一变量，无硬编码随机色值
□ 响应式：至少检查 mobile(375px) / tablet(768px) / desktop(1280px)
□ 无 accessibility 问题（图片有alt、按钮有aria-label）
□ 文件大小合理（单文件不超过500行）
```
